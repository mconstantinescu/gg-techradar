About Decision Records

The Decision Records section captures the "why" behind our technology choices. It houses two types of records:

  • Technology Decision Records (TDR) — enterprise-wide mandates set by the Technology Advisory Group (TAG).
  • Architecture Decision Records (ADR) — project- or programme-level decisions made by team stakeholders.

The 6-Domain Taxonomy

Decision records are organised into six domain-aligned categories:

Platform & Infrastructure

Compute, networking, cloud providers, Kubernetes, HA/DR.

Application Architecture

Integration patterns, API strategy, microservices vs. monolith.

Data & Analytics

Persistence, event streaming, AI/ML models, data governance.

Security & Compliance

Identity (IAM), encryption, Zero Trust, regulatory controls.

Development Experience

Languages, frameworks, CI/CD pipelines, developer portals.

Operations & Reliability

Observability, SRE practices, FinOps, incident management.

The 4-Stage Lifecycle (Rings)

Ring Stage Meaning
Standard Golden Path Recommended for all new workloads. The default choice unless a documented exemption exists.
Trial Validate Low-risk production use. Limited to specific projects or early-adopter teams.
Assess Research Under research or proof-of-concept. No production impact allowed yet.
Sunset Retire Formally deprecated. No new implementations. Migration plan required.

Governance Model

TDR — Technology Advisory Group (TAG)

  • Scope: Enterprise / Global
  • Binding: All new services; existing services must comply on major version change or within 12 months
  • Examples: Primary cloud provider, enterprise identity provider, CI/CD platform

ADR — Project Stakeholders & SMEs

  • Scope: Project or Product/Programme
  • Anchored in: Architecturally Significant Requirements (ASRs)
  • Examples: Event-driven order processing, service-level database choice

The Feedback Loop

  ┌─────────────────────────────────────────────────────┐
  │               TECHNOLOGY RADAR                       │
  │       (landscape of technologies we track)           │
  └──────────┬────────────────────────────┬──────────────┘
             │ informs                    ▲ updates
             ▼                            │
  ┌──────────────────┐        ┌───────────┴──────────────┐
  │  Project ADRs    │ pattern│  Enterprise TDR           │
  │  (team choices)  ├───────►│  (Global, TAG-decided)    │
  └──────────────────┘adoption└──────────────────────────┘
  1. Radar → ADR: A team evaluates a technology from the radar and writes an ADR to adopt it.
  2. ADR → TDR: When multiple projects independently adopt the same approach, the TAG formalises it as an enterprise TDR.
  3. TDR → Radar: The new TDR updates the radar, signalling endorsed status organisation-wide.

Contributing

To propose a new decision record:

  1. Create a markdown file in the latest edition folder of the Decisions repository.
  2. Set ring and segment in the frontmatter.
  3. TDRs require TAG review; ADRs require project stakeholder review.
